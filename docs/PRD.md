# 【猎鹰】智能招聘管理系统 - 产品需求文档 (PRD)

## 1. 项目背景与目标
*   **痛点：** HR 从 Boss 直聘等平台批量下载简历后，面临文件名乱码（Boss 特有乱码）、简历与作品集关联断裂、简历初筛工作量大、人岗匹配主观性强。
*   **目标：** 通过 AI 自动化解析与关联算法，实现从“乱码压缩包”到“结构化人才报告”的全链路处理。
*   **核心指标：** 提升初筛效率 80%，简历重命名准确率 95% 以上。

---

## 2. 核心业务流程 (Workflows)
1.  **定义标准：** HR 创建职位，AI 生成或录入 JD，系统将其转化为结构化“匹配基准 (JSON)”。
2.  **上传处理：** HR 拖拽上传简历 ZIP 包（包含简历 PDF 及作品集）。
3.  **智能分拣：** 系统自动解压，通过 PII（个人身份信息）算法关联简历与作品，并进行物理重命名。
4.  **AI 画像评分：** AI 解析简历内容，对比匹配基准，生成五维评分报告、履历核验及面试建议。
5.  **人工审核与导出：** HR 在工作台核对结果，修正少量误判，最后批量导出重命名文件或人才列表。

---

## 3. 功能模块详述 (Feature List)

### 3.1 职位管理 (Job Management)
*   **创建与基准定义：** 支持 AI 生成 JD，并自动转化为可编辑的 **结构化匹配基准 (JSON)**。
*   **权重调整：** 用户可手动微调技能项的权重（如：设置 Python 为“必要”或“加分”）。

### 3.2 智能分拣引擎 (Sorting Engine) - [核心难点]
*   **乱码处理逻辑：** **绝对不依赖文件名**，后端必须读取 PDF/Word 文本流提取特征。
*   **PII 自动关联：**
    1.  **提取：** 识别文本中的手机号、邮箱、微信号。
    2.  **关联：** 具有相同 PII 特征的文件自动识别为同一候选人的资产。
*   **物理重命名：** 自动将文件重命名为标准格式：`[姓名-岗位-文件类型].pdf`。

### 3.3 简历画像与逻辑核验
*   **断层识别：** 自动识别 >3 个月的履历空白期并在报告中显著标记。
*   **稳定性分析：** 自动计算平均在职时长。

### 3.4 人岗匹配评分系统 (Matching Score)
*   **五维评分：** 综合硬性条件、专业背景、稳定性、软技能、期望契合度；LLM 语义评分失败自动降级为规则式评分。
*   **面试助手：** 由 LLM 针对最低分维度生成 3 条面试题，每条附带「考察意图」说明；LLM 不可用时回落到模板题库。
*   **简历画像增强：** 简历文本结构化由 LLM 优先抽取（姓名/手机/邮箱/期望/技能/工作经历/教育经历），异常时自动回落到正则解析。

### 3.5 分拣工作台 (Human-in-the-loop)
*   **手动纠偏：** 支持用户通过拖拽交互手动关联“孤立文件”到特定候选人。

### 3.6 AI 能力全景 (AI Capability Matrix)
系统在以下五个阶段调用大模型（Doubao-1.5-Pro-32K），均配备规则式降级，保证在 LLM 不可用时功能不中断：

| # | 能力 | 触发点 | 输出 | 降级策略 |
| :-: | :--- | :--- | :--- | :--- |
| 1 | AI 生成 JD | 职位创建页点击「AI 帮我写 JD」 | 完整 JD 文案（不落库，供 HR 编辑） | 不降级，直接报错提示 |
| 2 | JD → 结构化匹配基准 | 保存 JD 后或点击「AI 解析」 | `JobCriteria` JSON（学历/年限/技能/软技能/薪资/地点） | 回落到正则关键词解析 |
| 3 | 简历文本 → 结构化画像 | ZIP 分拣后进入画像流水线 | `ResumeProfile` JSON（基本信息/经历/教育/技能/期望） | 回落到正则 `parse_resume` |
| 4 | 五维语义评分 | 画像 + 基准生成候选人报告时 | 五维得分与逐项解释（硬性/专业/稳定性/软技能/期望） | 回落到规则式 `score_candidate` |
| 5 | 面试提纲生成 | 报告生成后 | 3 条针对最弱维度的面试题 + 考察意图 | 回落到模板题库 `generate_questions` |

---

## 4. 页面布局定义 (UI/UX)
1.  **Dashboard：** 任务进度展示、高分人才雷达。
2.  **职位配置页：** 「AI 帮我写 JD」可折叠面板（输入框 + 生成按钮） + JD 文本编辑器 + 「AI 解析」按钮 + 结构化匹配基准预览/微调面板。
3.  **分拣工作台：** 三栏布局（左侧列表、中间预览、右侧纠偏与解析结果）。
4.  **候选人详情页：** AI 评分报告可视化（雷达图、时间轴）。

---

## 5. 项目结构与技术约束 (Technical Constraints)
*   **开发模式：** Monorepo (单仓)
    *   **前端目录：** `/frontend` (Next.js 14, Tailwind CSS, Shadcn UI)
    *   **后端目录：** `/backend` (FastAPI, PostgreSQL, PyMuPDF)
*   **AI 逻辑：** 语义化评分模型，非关键词命中统计。
*   **数据安全：** 文件物理隔离存储，支持 30 天自动清理机制。

---

## 6. 异常场景处理
*   **解析失败：** 若 PDF 为图片或加密，需在 UI 标记并引导人工处理。
*   **PII 缺失：** 降级采用“目录层级关联”及“文件名相似度算法”。

---

### 给 AI 助手的提示 (Important):
> “在实现此项目时，请严格遵守 `/frontend` 和 `/backend` 的目录划分。核心算法必须优先解决‘乱码文件名下的文件关联’问题，相关逻辑请参考 `docs/TDD.md`。”

---

### 为什么要这样改？

1.  **对齐目录命名：** 明确提到了 `/frontend` 和 `/backend`，防止 Augment 把代码写在根目录下。
2.  **明确单仓模式：** 让 AI 知道这是一个整体项目，在写 API 调用时会自动寻找后端定义。
3.  **统一逻辑：** 确保 PRD 里的 PII 提取、五维评分等专有名词与 TDD 完全一致。
